# SALT: Step-level Advantage Assignment for Long-horizon Agents via Trajectory Graph

- **ID**: 2510-salt
- **Link**: [arxiv](https://arxiv.org/pdf/2510.20022)
- **Authors**: Jiazheng Li, Yawei Wang, David Yan, Yijun Tian, Zhichao Xu, Huan Song, Panpan Xu, Lin Lee Cheong
- **Tags**: #credit-assignment #trajectory-graph #long-horizon #plug-and-play

## TL;DR

把**同 prompt 下的多条轨迹**构造成一个 **graph**（不仅仅是树），在 graph 上量化每个 step 的质量并分配 advantage。Plug-and-play 兼容 GRPO 等 group-based RL，**不改 rollout、不加计算开销**。在 WebShop / ALFWorld / AppWorld 上一致提升。

💡 **一句话精华**：同 prompt 多 rollout → trajectory graph（含 DAG 合并）→ 图上量化 step 质量派生 advantage，零 rollout 改动叠加 GRPO。

## Method

### 核心思路（根据 abstract）

现有问题：outcome reward 下，"有益 action" 和 "有害 action" 在 multi-step 交互里**纠缠在一起**——整条轨迹共享一个 return，好坏动作的 credit 分不开。

SALT 的做法：
1. 对同一 prompt 的所有 rollout 构造 **trajectory graph**
2. Graph 上量化每个 step 的质量
3. 据此分配 step-level advantage

**关键特征**：
- 只靠 outcome reward 推导 step-level advantage，无需额外 RM 或 MC 估值
- 不修改 rollout 流程
- 无额外计算开销
- Plug-and-play（可叠加 GRPO）

### 与相邻工作的关系

| 方法 | 结构 | 信号源 |
|---|---|---|
| GiGPO [2505-gigpo](2505-gigpo.md) | 同状态聚类（隐式图） | Outcome return |
| TreeGRPO [2509-treegrpo](2509-treegrpo.md) | 显式树 | Outcome return |
| SALT | **通用 graph**（含 DAG 合并） | Outcome return |

SALT 的 "graph" 比 tree 更一般：允许不同轨迹的**中间状态合并**（DAG），理论上能捕获更多的 credit 重叠。

### 🧬 Delta from GiGPO / TreeGRPO

- **GiGPO**：隐式图（同状态聚类），(action, return) 对在相同状态 group 内归一化
- **TreeGRPO**：显式 tree，但树的分支从 root 展开、不合并
- **SALT**：**通用 graph 含 DAG 合并**——不同轨迹的中间状态如果等价可以合并为同一节点，合并处 credit 多路汇聚

### 🧩 因果链（基于 abstract 推理）

- **问题**：outcome reward 下好 action 坏 action credit 纠缠在 multi-step 交互里
- **根因**：trajectory 共享 return，无机制区分贡献者
- **解法**：同 prompt 多 rollout 构 trajectory graph → 在 graph 上量化每 step 质量 → 派生 step-level advantage
- **效应**：WebShop / ALFWorld / AppWorld 一致提升（abstract 未给具体数字）

### ⚠️ What would break this (推测)

- **状态等价判定**：graph 边 / 节点合并的等价性阈值未明说；过严 → 退化为 tree，过松 → 合并错位
- **Rollout 过少时 graph 稀疏**：只有 1-2 条轨迹时 graph 无合并，失去优势
- **长 horizon 下图爆炸**：状态组合指数增长，维护成本可能高

## Key Results

## Takeaway

1. **"trajectory graph" 是比 tree 更一般的抽象**——值得关注它怎么合并中间状态
2. **Plug-and-play 的工程友好性**：如果 SALT 真能不改 rollout 就上，采纳成本很低
